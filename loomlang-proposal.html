Premise

The agent-first language probably isn't a nicer Rust, stricter Python, or LLM-friendly TypeScript — it's a typed artifact graph + capability-secure runtime + workflow language + evidence ledger + compiler to existing codebases. A language / runtime / repository / OS hybrid.

TL;DR

Human-first languages optimize for readable source text. Agents need something different: mechanically discoverable context, typed interfaces, bounded permissions, replayable execution, semantic diffs, provenance, and durable long-running workflows. The winning agent-first substrate isn't a syntax — it's a typed graph of intent, code, tests, traces, and evidence that compiles down into the languages we already have.

The central insight

The wrong question is "can a human read this file?" The right one is can an agent safely change this, prove what changed, recover from failure, and explain the result to a human reviewer? That's a different design center.

Optimizes for human authorship

Human-first languages

  • Readable source text
  • Local reasoning
  • Expressive syntax
  • Flexible abstraction
  • Interactive debugging
  • "I can understand this file"
Optimizes for agent delegation

Agent-first substrate

  • Mechanically discoverable context
  • Explicit invariants & typed interfaces
  • Bounded permissions, replayable execution
  • Semantic diffs, provenance, evidence
  • Durable long-running workflows
  • Verifier-friendly, compact for context windows

Where the claim is strong — and where it overreaches

The strongest version is not "all languages disappear." It is: existing languages were designed around human authorship, and agent-heavy production will expose that mismatch so severely that a new substrate becomes economically dominant.

But programming languages won't converge the way operating systems did — they serve too many roles. Expect convergence at the agent-native intermediate layer, not the surface syntax. Useful analogies:

The universal work substrate — what agents read, mutate, verify, and compile

1. The source of truth is a typed artifact graph, not text files

The canonical program is not a directory of files. It is a graph of typed artifacts — each with type, owner, version, dependencies, invariants, permissions, provenance, failure modes, test obligations, and compiler targets.

A notes.md file titled "decisions" is weak. A typed Requirement linked to AcceptanceCriteria, TestCase, CodePath, Migration, and EvidenceBundle is dense. Unison points at this with content-addressed code; LoomLang generalizes the move — requirements, tests, traces, decisions, and proofs are all content-addressed too.

2. Every change is intent-carrying

No implementation change exists without an associated change object. This isn't documentation — the compiler reads it to generate obligations.

change AddInvoiceCsvExport {
  intent:
    "Allow billing admins to export filtered invoices as CSV."
  user_value:
    role: BillingAdmin
    outcome: "download invoice data for reconciliation"
  acceptance:
    - "Export respects active filters"
    - "CSV includes invoice_id, customer_id, amount, status, due_date"
    - "No PII beyond fields already visible in billing table"
    - "Export is audit-logged"
  risk:
    data_sensitivity: internal_financial
    requires_security_review: true
  evidence_required:
    - unit_tests
    - integration_test
    - permission_test
    - audit_log_test
    - ui_trace
    - rollback_plan
}

Touches financial data → permission checks. Alters a schema → migration plus rollback. Touches UI → UI trace. Changes a public API → compatibility checks. "Done" becomes impossible to claim without evidence.

3. Types include effects, capabilities, permissions, and risk

A traditional type system says String -> Int. Agent-first types need to say much more:

function:
  input:  InvoiceFilter
  output: CsvFile
  effects:
    - db.read[Invoice]
    - audit.write[ExportEvent]
    - file.generate[Csv]
  forbidden:
    - db.read[Customer.ssn]
    - network.external
  requires:
    - role[BillingAdmin]
    - pii_policy_check
  evidence:
    - test.permission_denial
    - test.audit_log_created
No ambient authority

Agents should not operate in systems where any code can reach the network, filesystem, or database. They should be forced to declare what they need, and the runtime should grant only those capabilities. WASI points at this for execution; LoomLang generalizes it to every database, secret, deployment target, and user-data access.

4. Programs are durable workflows by default

Agents don't just write functions. They run long tasks: reproduce, localize, patch, test, review, package, deploy, monitor, roll back. That isn't a function call. It's a durable workflow — and the language should treat workflows as first-class.

workflow FixBug(report: BugReport) -> EvidenceBundle {
  step reproduce using BrowserHarness
  step localize  using TraceSearch
  step patch     using CodeEdit
  step validate  using TestSuite
  step review    using AgentReview[security, architecture]
  step package   using PullRequest
  step monitor   using CanaryMetrics

  on failure:
    preserve_trace
    classify_failure
    propose_next_action

  on human_required:
    emit DecisionRequest
}

Fuses three traditions: Temporal's durable execution with deterministic replay, P's communicating state machines with checkable correctness properties, and ordinary code's familiarity — plus first-class typed escalation to humans.

5. Two representations — machine-canonical, human-explainable

The canonical representation doesn't need to be hand-written text. It can be a graph IR, an AST database, content-addressed objects, or binary components. But the system must always produce reliable human projections:

Q1

What changed?

Behavior-level summary, not file edits.

Q2

Why did it change?

Linked intent & user value.

Q3

What evidence?

Tests, traces, screenshots, logs.

Q4

What risks remain?

Open invariants & assumptions.

Q5

What rollback exists?

Reversibility plan, executable.

Q6

What was discarded?

Branches the agent tried and rejected.

Today, source code is primary and explanations are secondary. In LoomLang the graph is primary; code and explanations are projections.

6. Semantic diffs matter more than textual diffs

What we have today

Textual diff

+ if user.is_admin:
+     allow_export()

Reviewer must reverse-engineer intent and risk from the lines.

What LoomLang surfaces

Semantic diff

Capability added:
  billing.export_csv
Permission boundary:
  BillingAdmin → CsvFile of Invoice
Data exposure:
  invoice_id, customer_id, amount,
  status, due_date · no new PII
Audit:
  new ExportEvent on success
Tests added:
  + admin_can_export
  + non_admin_denied
  + export_is_audit_logged
Residual risk:
  CSV not load-tested > 100k rows

Agents operate on semantic diffs. Humans review semantic diffs. Textual diffs still exist — they live a layer below.

7. Testing is not separate from programming

Tests aren't files in a folder — they are obligations generated by artifact types. The compiler knows an endpoint touching sensitive data requires permission tests; a migration requires rollback tests; a UI state requires a DOM trace.

api POST /invoices/export {
  requires role BillingAdmin
  reads    Invoice
  returns  CsvFile
  obligations:
    - permission_denial_test // for non_admin
    - audit_log_test         // for success
    - property_test          "exported rows match filter"
    - regression_test        // from prior bug reports
}

Agents are very good at producing plausible code. The substrate forces them to produce verifiable work packets — making EvidenceBundle as fundamental as a function return type.

8. Context discovery is built into the language

Today, agents waste enormous effort asking: where is the code? what command runs tests? which docs are authoritative? what does "done" mean here? LoomLang declares this as a typed, queryable, enforced context block:

context BillingExport {
  canonical_docs:
    - BillingDomainModel
    - InvoicePermissionPolicy
    - CsvExportPattern
  relevant_components:
    - InvoiceRepository
    - BillingTableUI
    - AuditLogService
  forbidden_patterns:
    - direct_sql_from_ui
    - unaudited_download
    - customer_pii_in_csv
  validation_commands:
    - test BillingExportSuite
    - lint DependencyBoundaries
    - eval CsvExportGoldenCases
}

Effectively a machine-readable AGENTS.md — typed, linked to code, mechanically enforced.

9. The compiler is partially self-healing

A normal compiler says Type error at line 43. An agent-first compiler should expose a structured repair interface:

Violation:
  BillingExportUI depends directly on InvoiceRepository.
  Domain rule forbids UI → Repo dependency.

Allowed path:
  UI → Runtime → Service → Repo

Suggested repair:
  Move export call into BillingExportService.
  Expose typed command BillingExportRuntime.requestCsvExport.

Generate patch? [y/n]

And every failure can be promoted into the language itself:

failure PatternViolation_2026_05_18 {
  cause:    direct_repo_access_from_ui
  repair:   introduce_service_boundary
  promoted_to_rule: DependencyBoundaryLint
  regression_guard: enabled
}

That's how an agent factory accumulates institutional memory. Every failure becomes a lint, a test, a policy, a refactoring recipe, an eval example.

10. Native notion of uncertainty

Human-written programs pretend uncertainty doesn't exist. Agent-written programs cannot. LoomLang tracks epistemic state directly:

claim "Existing export endpoint is unused" {
  confidence: 0.72
  evidence:
    - no references in repo_search
    - no route hits in last_30_days_logs
  missing:
    - production dashboard unavailable
  action:
    require human_confirmation before deletion
}

The language distinguishes between proven tested observed inferred guessed assumed human-approved — categories normal languages have no place to put.

11. First-class human escalation

Not "ask the user." A typed, structured decision request that lets the agent continue everything else while marking one specific decision as blocked.

decision_request DataRetentionAmbiguity {
  question:
    "Should exported invoice CSVs be stored server-side or generated ephemerally?"
  options:
    A: "Ephemeral generation only"
    B: "Store generated CSV for 24 hours"
    C: "Store export history but not file contents"
  recommendation: A
  rationale:
    - lower data retention risk
    - simpler compliance posture
    - sufficient for current user story
  blocked_steps:
    - finalize ExportStoragePolicy
}

Human work becomes exception handling, not babysitting.

12. Compiles to existing languages

The winning agent language doesn't ask the world to be rewritten. It sits above existing repos as a meta-language — and emits ordinary code.

component BillingExport
  target backend:  python.fastapi
  target frontend: typescript.react
  target infra:    terraform
  target tests:    pytest + playwright

The agent works in LoomLang. The repo still receives ordinary code. Adoption doesn't require rewriting anything.

A taste of the syntax

A small module — describing user intent, data model, permissions, effects, invariants, tests, workflow, and done condition in one artifact:

module Billing.ExportCsv

goal AddInvoiceCsvExport {
  actor:   BillingAdmin
  outcome: "Export visible invoices to CSV for reconciliation"
  risk:    internal_financial_data
}

resource Invoice {
  fields:
    invoice_id:     PublicId
    customer_id:    CustomerId
    amount:         Money
    status:         InvoiceStatus
    due_date:       Date
    internal_notes: SensitiveText @not_exportable
}

capability CsvExport {
  allowed_roles: [BillingAdmin]
  reads:         Invoice[visible_to_actor]
  writes:        AuditLog.ExportEvent
  forbids:
    - Invoice.internal_notes
    - Customer.ssn
    - network.external
}

service ExportInvoices {
  input:  InvoiceFilter
  output: CsvFile
  effects:
    - db.read[Invoice]
    - audit.write[ExportEvent]
    - file.generate[CsvFile]
  invariants:
    - exported_rows == invoices_matching(input)
    - no_field(classified_as: restricted_pii)
    - audit_event.exists for every successful_export
  obligations:
    - unit_test         rows_match_filter
    - permission_test   non_admin_denied
    - property_test     no_restricted_fields
    - integration_test  csv_download
    - ui_trace          billing_admin_exports_csv
}

workflow ImplementChange {
  steps:
    - inspect_existing_billing_patterns
    - generate_service
    - generate_ui_action
    - generate_tests
    - run_obligations
    - produce_evidence_bundle
    - open_pull_request
  stop_when:
    all_obligations_pass
    and semantic_diff.reviewable
}

Notice what this describes: not "the code that exports CSVs", but the contract under which an agent is allowed to produce that code, and the evidence required to call it done.

The ideal type system

The type system layers ordinary static types with several agent-native dimensions — each captures something today's languages either ignore or scatter across comments, lint rules, and tribal knowledge.

Static

Ordinary correctness — InvoiceFilter → List[Invoice].

Refinement

Value constraints — Percent where 0 ≤ value ≤ 100, CsvFile where rows ≤ max_export_rows.

Effect

What the code can do — db.read[Invoice], audit.write[ExportEvent], network.none.

Capability

What authority is granted — requires role[BillingAdmin], forbids secret[ProductionDatabaseWrite].

Resource

For data, money, files, irreversible operations — ExportedCsv is EphemeralFile expires_after 10 minutes, PaymentCapture is IrreversibleAction.

Risk

Useful for governance — low · customer_visible · regulated_data · safety_critical.

Evidence

A change is only complete if it returns the required evidence — Change[BillingExport] → EvidenceBundle[UnitTests, PermissionTests, AuditLogTrace, UiTrace, RollbackPlan].

Types stop being only documentation. They become work contracts.

A normal runtime asks

"How do I execute this program?"  An agent-first runtime asks: "How do I let uncertain model-driven workers safely modify, test, explain, and deploy a system over time?" That is much more OS-like than VM-like.

What to build first

The practical MVP is not a whole new language. It's a repo-local agent substrate that sits above existing code — usable today on a Python or TypeScript monorepo, without inventing syntax.

Piece 1

WorkSpec.yaml

Typed task object: intent, scope, risks, acceptance criteria, affected domains, required evidence.

Piece 2

ArtifactGraph

Index code, docs, tests, APIs, schemas, commands, policies, prior PRs as typed nodes.

Piece 3

CapabilityPolicy

Declare what agents may read, write, execute, access, deploy.

Piece 4

EvidenceBundle

Standard output: diff, tests, traces, screenshots, logs, semantic summary, residual risks.

Piece 5

SemanticDiff

Explain behavior-level changes, not file edits.

Piece 6

ObligationCompiler

Given a WorkSpec, generate required checks.

Piece 7

AgentRepairProtocol

Convert failures into structured repair hints.

Piece 8

HumanDecisionRequest

Typed escalation instead of vague "what should I do?"

A concrete MVP — WorkSpec.yaml
workspec:
  id:     billing.export_csv
  intent: Allow billing admins to export visible invoices as CSV.
  actor:  BillingAdmin

  scope:
    include: [billing-ui, billing-service, invoice-repository]
    exclude: [payment-processing, customer-profile-service]

  risk:
    data_classification: internal_financial
    customer_visible:    true
    regulated:           false

  capabilities:
    allow:
      - read:     invoices
      - write:    audit_logs
      - generate: csv_file
    deny:
      - read:     customer_ssn
      - network:  external
      - write:    payments

  acceptance:
    - Export respects active filters.
    - Non-admin users cannot export.
    - Export is audit logged.
    - CSV does not include restricted PII.

  evidence_required:
    - unit_tests
    - integration_tests
    - permission_tests
    - ui_trace
    - semantic_diff
    - rollback_plan

That's the first step toward the language Geoff is imagining. The agent isn't told "add CSV export". It is given a typed work contract.

Common pushback

"Don't dynamic languages disappear under this?"

No — but untyped production surfaces become liabilities. Python in a notebook stays useful. Python without typed boundaries, schemas, tests, and artifact metadata becomes a legacy island inside a typed agent substrate. The bet is on wrapping, not killing.

"Why not just keep iterating on AGENTS.md?"

AGENTS.md is a great first step. It's also unenforced markdown. The moment context goes stale, agents quietly diverge. LoomLang's bet is that the substrate must be typed and machine-checked, not narrative — same conceptual move as JSON Schema vs. "the API docs in Confluence".

"Isn't this just a workflow engine plus a type checker?"

That's roughly the right shape. The hard part isn't any one of them — it's the composition: a workflow whose steps emit typed evidence, whose effects are gated by capability policy, whose diffs are semantic, whose failures become future lints. Each piece exists; the integration is what's missing.

"What about the people who want to read the code?"

They get a much better artifact than a 4,000-line diff. They get a semantic diff, the linked intent, the evidence bundle, the residual risks, and (if they want) the underlying generated code. The human-readable layer becomes a projection — but a denser, more honest projection than today's PR description plus 28 changed files.

Programming as constrained delegation

Human-first programming is "I write instructions for a machine."

Agent-first programming is "I specify goals, constraints, permissions, and evidence obligations for model-driven workers operating inside a controlled software substrate."

Final form

A statically typed, capability-secure, content-addressed, workflow-native artifact graph that compiles human intent into code, tests, docs, policies, deployments, traces, and evidence bundles — while preserving enough semantic structure for agents to safely modify it and humans to review it.

Or more simply: a language where every change carries its intent, authority, proof obligations, execution trace, and explanation.

That's the thing that could plausibly cause a convergence event — not because humans stop caring about programming languages, but because agents need a different substrate, and once software production becomes agent-dominant, the substrate agents prefer becomes the substrate everyone else is forced to accommodate.
